# 新增动画效果模板流程

为 `templates/` 新增动画效果模板时，遵循本流程。**3 个阶段，每阶段末必须向用户呈现产出并等待确认后才进下一阶段；被拒则原地修订重提。**

> 本流程仅用于新增 `templates/` 下不存在的效果。已有效果的复用/转换仍走 `SKILL.md` 主工作流。

---

## 阶段① 定义

**目标**：把模糊的「我想加个 XX 效果」固化成一份可直接实现的效果定义卡，经用户确认后作为阶段②的唯一输入。

**步骤**：

1. **采集输入来源** — 问用户：「你有参考实现吗（HTML 文件 / 设计稿 / 参考网站）？」
   - 有参考 → 读取并分析，提取效果机制（CSS/JS 手法、数据结构雏形）
   - 无参考 → 基于关键词描述从零设计效果机制

2. **填写效果定义卡**（固定 8 字段）：

   | 字段 | 内容 | 说明 |
   |------|------|------|
   | 目录名 | `templates/<kebab-case>/` | 由效果名推导 |
   | 中文名 + PascalCase 标识 | 如「文字打字 (Typewriter)」 | 与现有命名一致 |
   | 关键词 | 中英文逗号分隔，含同义词 | 兼顾 SKILL.md 触发表 + README |
   | 分类 | 数据展示 / 交互展示 / 布局 / 文字动画 / … | 扩展现有分类 |
   | 描述 | 一句话效果 + 适用场景 | — |
   | 依赖 | 无 / 库名（注明与框架无关） | — |
   | 数据 schema | 字段表（字段/类型/必填/说明） | 从参考提取或设计 |
   | 可调参数 + 示例数据 | 参数表（参数/类型/默认/说明）+ 示例数据 JSON | 4-6 项 |

3. **AI 自检定义卡**（退出标准）：
   - [ ] 关键词不与现有 `templates/*/README.md` 关键词重复冲突
   - [ ] 数据 schema 每字段都有明确类型和说明
   - [ ] 示例数据 4-6 项、字段与 schema 表对齐
   - [ ] 分类属于已知集合或合理新增

4. **门控** — 向用户完整呈现效果定义卡，提示：
   > 以上是效果定义，确认后我开始实现 effect.html 和 README。需要调整哪里？

**被拒处理**：原地修订对应字段，重新呈现整张卡，不进入阶段②。

---

## 阶段② 实现

**目标**：按已确认的效果定义卡产出 `effect.html` + `README.md`，通过 AI 代码自查后交用户浏览器视觉自测。

**步骤**：

1. **实现 effect.html** — 遵循模板规范：
   - 完整 `<!DOCTYPE html>` + 内联 `<style>`（纯 CSS）+ 内联 `<script>`（原生 JS）
   - 无外部资源依赖；图片用 `https://picsum.photos` 占位
   - 示例数据用 `data-*` 或语义结构标注，让 AI 一眼看出「这是数据」
   - 可调参数在 `<script>` 顶部用常量声明并注释「可调」
   - 双入口差异：有参考则清洗移入（去冗余、统一结构）；无参考则从零编写

2. **实现 README.md** — 框架中立格式：
   - 标题 / 关键词 / 分类 / 描述 / 依赖
   - 数据结构表（字段/类型/必填/说明）—— 与定义卡 schema 一致
   - 可调参数表 + 示例数据 JSON
   - 转换提示固定一句指向 `component-spec.md`

3. **AI 代码自查**（退出标准，逐条核对）：
   - [ ] `effect.html` 浏览器可直接打开，无外部本地依赖
   - [ ] README 数据结构表每个字段都能在 effect.html 找到对应 `data-*` 或结构标注
   - [ ] 可调参数在 effect.html 顶部可配置且默认值与 README 表一致
   - [ ] 含响应式断点（`@media`）
   - [ ] 含 `prefers-reduced-motion` 降级处理
   - [ ] README 关键词与定义卡一致
   - [ ] 目录名、分类、依赖与定义卡一致

4. **门控** — AI 呈现：自查报告（逐条 ✓/✗）+ effect.html 与 README 路径。提示用户：
   > 请浏览器打开 effect.html 确认视觉效果。通过后我进入集成阶段。

**被拒处理**：
- AI 自查有 ✗ → 原地修复后重提自查报告
- 用户视觉不满意 → 按反馈调整 effect.html（不改定义卡 schema，除非用户要求），重新自查 + 重交用户自测

---

## 阶段③ 集成

**目标**：把已通过自测的新模板接入技能索引层，确保能被关键词匹配和 evals 覆盖。

**步骤**：

1. **更新 SKILL.md 三处**：
   - frontmatter `description` — 追加新效果关键词（如「打字效果/逐字动画」）
   - 触发识别表 — 新增一行（效果 / 关键词 / 用户想要什么）
   - 可用效果一节 — 新增小节（目录 / 效果 / 依赖）

2. **更新 evals（默认加，可跳过）**：
   - 新增一条 `evals/evals.json` 用例：`prompt`（用户口吻描述需求）+ `expected_output`（应匹配该模板，按探测框架转换，遵循 component-spec.md）
   - 用户可明确跳过；跳过时在本次执行记录里注明原因（如「内部演示效果，不纳入回归」）

3. **落地提示** — 向用户呈现集成清单：
   - 新增/修改的文件列表
   - 新效果的关键词（提醒用户日后如何触发）
   - 若跳过了 evals，注明跳过原因

4. **门控** — 提示用户：
   > 集成完成。新增效果 X，关键词：…。确认无误则本次新增结束。

**被拒处理**：按反馈修正对应文件（如关键词漏了同义词、evals prompt 不够自然），重新呈现集成清单。
